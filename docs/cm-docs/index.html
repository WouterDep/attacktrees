Countermeasure description

This table enlists all defined countermeasures with a description, their complexity to implement and difficulty increase on the node it's applied to.

Nr.CM nameDescriptionComplexityDifficulty increase
1Alert/delay on unsuccessful loginWhen a series of unsuccesful logins exceeds a certain limit, generate an alert and/or delay the next login.LOW1
2Application monitorInstall an application monitor that evaluates the content of the commands and messages using Deep Packet Inspection (DPI). Any suspicious commands are reported.MEDIUM2
3Avoid IP-address rangesAvoid ranges of IP-addresses in the firewall rules. An unused but allowed IP-address can be used to sneak through the firewallMEDIUM1
4Change default passwordsA lot of default passwords are published in the vendors documentation.LOW1
5Change firewall policyRestrict access from this component to the network if unnecessary.MEDIUM2
6Checksums on msgsAdd checksums to the messages (or even better: encrypt/hash messages using secret key)MEDIUM1
7Data diodeA data diode physically restricts data flow to only one direction. This provides a physical segmentation with good performance and few maintenance.LOW4
8Disable LM hashDisable LM hashes (on all Windows versions before Vista and Server 2008), because these hashes can be cracked really fast. Prevent automatic generation of LM hash by using a password of at least 15 characters.MEDIUM1
9Disable unused ports & servicesUnused ports and services are a common attack vector. An unused service can contain vulnerabilities, unused ports can have unintended access to other components in the network. Disable both if unnecessary.MEDIUM2
10Encrypted tunnelExchange data within the ICS through an encrypted tunnel. Avoid FTP, telnet and rlogin. Any communication protocol can be 'tunneled' through SSH. Use HTTPS instead of HTTP.MEDIUM2
11Input validation: bounds checkingMake the application resilient for abnormal values: very large, negative, zero, floating point, ... This can often cause a crash of the system if it can't handle valid values.MEDIUM2
12Input validation: buffer lengthPerform a code review of applications that handle network traffic. Check for buffer overflow vulnerabilities by implementing input validation. Inspect the lenght of the input and buffer (esp. for C/C++).MEDIUM1
13Input validation: code injectionPrevent Code Injection by creating statements statically, whitelist of acceptable inputs, reject if not valid or transform. Use prepared statements to avoid SQL injection.MEDIUM2
14Input validation: Path TraversalPrevent Path Traversal by defining whitelist of acceptable inputs, reject if not valid or transform. Use prepared statements to avoid SQL injection.MEDIUM2
15Install Anti-VirusPrevent installation of malware (Trojan Horse, keylogger, ...) on the workstation of an employee.LOW2
16Install IDS/IPSIntrusion Detection and Prevention systems inspect the network traffic for malware or exploits. IDS alerts suspicious packets, IPS can also drop them, preventing further harm. Some available IDS/IPS devices have anomaly detection functionality. Using statistical analysis of common behaviour, unusual events are detected, might mark unknown threats.HIGH1
17Install IDS/IPS & Protocol Anomaly Detect.Intrusion Detection and Prevention systems inspect the network traffic for malware or exploits. IDS alerts suspicious packets, IPS can also drop them, preventing further harm. Monitor network traffic for changing MAC addresses to prevent MitM attacks. Choose for a device with Protocol Anomaly detection.HIGH2
18Install updates/patchesInstall software updates. Vulnerable elements will be protected to newly discovered vulnerabilities.HIGH3
19Least privilegeApply the Principle of Least Privilege: Grant an employee only the minimum privileges required to perform his job.MEDIUM2
20Physical IsolationStrong physical isolation (fence, heavy doors) and strict authentication (doors with electronic loc), surveillance (guards, CCTV)LOW3
21Physical system hardeningObstruct unused ports in order to make physical connection impossible.LOW2
22Port SecurityCheck MAC address of connected devices and refuse access if unknown device. Disable unsecured/unused ports. Disable USB ports to prevent malware injection.LOW2
23Restrict outbound trafficAn attacker needs feedback during his attack. If outbound traffic is restricted appropriately, the attacker cannot establish an outbound connection for feedback, and is therefore limited to blind attacks.HIGH2
24Static ARP table/Dynamic inspectionUse hard-coded ARP tables for static IP addresses or dynamic ARP inspection for dynamic IP addresses to prevent MitM attacks.HIGH1
25Strong PasswordsUse strong passwords with an expiration time. (See recommendations specified in ISATR99.00.02-2004).LOW1
26VPN for remote connectionUse Virtual Private Network for remote connection to the enterprise. With an unencrypted channel, the credentials could be sniffed.MEDIUM1
